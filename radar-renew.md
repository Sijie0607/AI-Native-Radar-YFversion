# 雷达自动更新｜产品 Brief 定稿版

## 1. 功能背景

- AI-Native 读书雷达当前已支持用户浏览书籍、查看详情、推荐书籍，以及对已有书籍进行评分。
- 当前评分提交后，推荐指数会实时更新，但首页雷达中的书籍视觉表现不会同步反映变化。
- 本轮新增“雷达自动更新”能力，让推荐指数变化立即体现在雷达图上，但不破坏“领域 + 难度”这套基础阅读逻辑。

## 2. 本轮目标

- 用户评分后，推荐指数实时更新
- 雷达对推荐指数变化做即时反馈
- 保持 8 大领域和 3 层难度结构稳定
- 不让推荐指数和难度语义混淆
- 让用户更容易识别“哪些书更值得优先关注”

## 3. Scope

### 3.1 本轮要做

- 用户评分后，雷达立即自动更新
- 推荐指数变化体现在视觉权重上
- 保持 8 大领域分区不变
- 保持难度圈层归属不变

### 3.2 本轮不做

- 历史版本对比
- 同圈层内重排
- 账号体系
- 审核后台
- 真实后端
- 社区互动
- 自动收录
- 拖拽 / 缩放
- 跨圈层位置变化

## 4. To-Be Journey

1. 浏览雷达首页
2. 用户对书籍评分
3. 推荐指数实时更新
4. 雷达立即更新该书的视觉权重
5. 用户继续浏览并感知这本书在雷达中的变化

## 5. 已确认事实

### 5.1 雷达结构语义

- 扇区仍表示 8 大领域
- 圈层仍表示难度
- 这两个基础结构本轮不变

### 5.2 推荐指数表达方式

- 推荐指数本轮不改变书籍所属圈层
- 推荐指数本轮不做“越高越靠外”表达
- 推荐指数本轮只通过视觉权重变化表达
- 本轮不做同圈层内重排

### 5.3 视觉权重手段与优先级

- 本轮保留全部 3 种视觉手段：
  - `1.` 圆点大小
  - `2.` 光晕 / 描边
  - `3.` 颜色强度
- 优先级按以上顺序处理
- 即：
  - 大小是主表达
  - 光晕 / 描边是辅助强化
  - 颜色强度是最轻的补充表达

### 5.4 更新时机

- 本轮按“每次评分后立即更新”处理
- 即推荐指数变化后，雷达同步刷新

### 5.5 历史版本范围

- 本轮先不考虑历史版本
- 只做当前版的自动更新反馈

## 6. 与现有产品的关系

### 6.1 不冲突的部分

- 产品定位本来就强调“动态演化”
- 当前评分功能已提供推荐指数更新基础
- 当前首页雷达已具备浏览主流程，可以直接承接自动更新后的反馈

### 6.2 本轮新增的能力

- 推荐指数变化不再只体现在详情数值中
- 雷达会同步体现“这本书更值得关注了”
- 但不会破坏用户对“圈层=难度”的理解

## 7. Journey 断点判断

### 7.1 当前评分与雷达视觉反馈断开

- 现在分数会更新
- 但雷达没有同步视觉变化
- 本轮就是补这个断点

### 7.2 当前推荐指数表达过于单一

- 现状中更多只是数值变化，用户需要点开详情才能感知
- 本轮要让用户在首页雷达上直接感知变化

### 7.3 当前不能用位置变化表达分数

- 因为圈层已经承载难度语义
- 所以本轮明确不采用“外移”或“重排”方式
- 改为纯视觉权重变化

## 8. 进入 UI 设计前的固定业务规则

- 每次评分后立即触发雷达更新
- 8 大领域分区不变
- 难度决定圈层，不变
- 推荐指数不改变圈层归属
- 推荐指数只通过视觉权重变化表达
- 本轮保留圆点大小、光晕 / 描边、颜色强度三种手段
- 表达优先级：
  - 圆点大小
  - 光晕 / 描边
  - 颜色强度
- 本轮不做历史版本
- 本轮不做同圈层内重排

## 9. 已知事实 / 设计建议 / 待确认假设

### 已知事实

- 评分后推荐指数会实时更新
- 本轮雷达要立即自动更新
- 本轮不做历史版本
- 本轮不改变 8 大领域和难度结构
- 本轮不采用位置变化表达推荐指数

### 设计建议

- 圆点大小作为主表达应最明显，但不要大到破坏整体秩序
- 光晕 / 描边作为辅助信号，用于强调“值得关注”
- 颜色强度要克制，避免覆盖领域色本身的识别
- 更新反馈应专业、克制，不做夸张炫技效果

### 待确认假设

- 当前方向性假设已基本收敛
- 下一轮重点应进入：
  - 各种视觉手段的使用边界
  - 默认态 / 更新后态 / hover / 选中态如何共存
  - 用户如何理解“推荐指数提高了”

## 10. 改变 / 不变 / 复用 / 不做

### 改变

- 推荐指数变化后，雷达会立即响应
- 用户能在首页直接看到评分带来的影响
- 雷达中的书籍权重不再只是静态展示

### 不变

- 内容对象仍是书籍
- 8 大领域结构不变
- 难度圈层结构不变
- 首页雷达浏览仍是主流程
- 不引入拖拽和缩放

### 复用

- 复用当前评分功能产生的推荐指数
- 复用当前首页雷达结构
- 复用当前详情、列表与评分主流程
- 复用现有领域和难度基础模型

### 不做

- 不做历史版本对比
- 不做同圈层内重排
- 不做账号体系
- 不做真实后端
- 不做审核后台
- 不做社区能力
- 不做跨圈层位置变化

## 11. 一句话基线

- 本轮雷达自动更新的定位是：用户每次评分后，推荐指数立即驱动该书在雷达中的视觉权重变化，重点通过圆点大小、光晕 / 描边和颜色强度体现“更值得关注”，但不改变 8 大领域与难度基础结构。

## 12. 数据、接口与实现约束

### 12.1 当前真实代码基线

- 当前雷达组件已经使用 `recommendationScore` 影响圆点大小，但这只是单一尺寸映射，不等于完整的视觉权重系统。
- 当前已具备评分成功后更新书籍推荐指数与推荐人数的能力。
- 当前雷达点位 `x / y`、领域 `sectorIndex`、难度圈层 `ringIndex` 均为稳定结构，本轮不改。

### 12.2 数据模型原则

- 本轮不新增新的后端数据模型，不引入真实接口。
- 本轮不为 `Book` 增加持久化的视觉字段。
- 本轮继续以 `recommendationScore` 作为雷达视觉权重变化的唯一分值输入源。
- `x / y`、`sectorIndex`、`ringIndex`、`difficultyLevel` 继续保持不变。

### 12.3 前端衍生视觉模型

- 本轮建议新增一个纯前端衍生类型，用于统一描述雷达中单本书的视觉样式。
- 该衍生模型不写回书籍数据，只在渲染雷达时即时计算。
- 建议最小字段包括：
  - `pointRadius`
  - `haloRadius`
  - `haloOpacity`
  - `strokeWidth`
  - `strokeOpacity`
  - `fillOpacity` 或 `colorIntensity`
  - `zIndexPriority`
  - `isRecentlyUpdated`

### 12.4 最近更新状态

- 本轮需要一个短时 UI 状态，用于表示“刚刚评分成功并已同步到雷达”的书籍。
- 建议状态最小定义：
  - `recentlyUpdatedBookId`
  - `recentlyUpdatedAt`
- 该状态不属于持久业务数据，不写入 `Book`。
- 建议作为雷达视图层状态管理，而不是作为书籍字段保存。

### 12.5 数据流与接口建议

- 本轮不新增真实 API。
- 继续沿用现有评分提交流：
  1. 用户提交评分
  2. 评分服务返回最新推荐指数和更新后的书籍数据
  3. 主书籍 store 同步更新该书的 `recommendationScore`、`votesCount` 与 `recommendations`
  4. 同时标记该书为“最近更新”
  5. 雷达基于新的推荐指数重新计算视觉权重

- 本轮不需要新增：
  - 位置重算接口
  - 历史版本接口
  - 版本快照接口

### 12.6 视觉映射收口方式

- 建议新增一个前端纯函数，统一完成“推荐指数 -> 视觉权重”的映射。
- 该函数输入建议包括：
  - `recommendationScore`
  - `domainColor`
  - 当前交互态：默认 / hover / 选中 / recentlyUpdated
- 输出建议包括：
  - 点大小
  - 光晕半径与透明度
  - 描边宽度与透明度
  - 颜色强度
  - 层级优先级

### 12.7 页面状态与数据状态联动

- 默认态：
  - 每本书根据当前 `recommendationScore` 自动映射视觉权重。
- 评分成功后：
  - 推荐指数更新
  - 雷达立即重渲染
  - 对应书籍进入一次短暂更新反馈态
- 更新反馈态：
  - 仅作用于刚更新的书
  - 不改变位置
  - 只做视觉强化
- Hover 态：
  - 优先级高于默认高分态
- 选中态：
  - 优先级高于 Hover 和最近更新态

- 建议状态优先级：
  1. `selected`
  2. `hover`
  3. `recentlyUpdated`
  4. `defaultScoreWeighted`

### 12.8 视觉权重实现规则

- 本轮保留 3 种视觉手段，优先级如下：
  1. 圆点大小
  2. 光晕 / 描边
  3. 颜色强度

- 实现原则：
  - 圆点大小是主表达，变化最明显
  - 光晕 / 描边是辅助强化
  - 颜色强度是最轻的补充表达

- 建议先对推荐指数做归一化处理：
  - `normalized = (recommendationScore - 3) / 2`
  - 范围控制在 `0 ~ 1`

- 再分别映射为：
  - `pointRadius`
  - `haloOpacity`
  - `strokeWidth`
  - `fillOpacity`

### 12.9 颜色与动画约束

- 本轮不改变领域色相。
- 颜色强度只允许在同色系内做轻量强弱变化。
- 本轮不引入复杂动画库。
- 只使用现有 SVG / CSS 过渡能力完成：
  - `r`
  - `stroke-width`
  - `opacity`
  - 光晕相关视觉属性

- 更新反馈动画必须：
  - 时间短
  - 变化轻
  - 不夸张
  - 不影响整体可读性

### 12.10 组件与模块影响范围

- 本轮核心改动组件：
  - `RadarChart`
  - 评分结果组件
  - 首页承载评分完成返回后的联动状态

- 建议新增一个视觉映射工具模块，用于统一维护视觉权重公式。

- 本轮不应改动：
  - 雷达点位分布算法
  - 领域划分结构
  - 难度圈层结构
  - 推荐书籍业务规则
  - 评分业务规则

### 12.11 改变 / 不变 / 复用

#### 改变

- 雷达中的书籍点将根据推荐指数实时映射视觉权重。
- 评分成功后，首页雷达会对对应书籍给出一次短时更新反馈。
- 推荐指数不再只在详情中可感知。

#### 不变

- `x / y` 不变
- 圈层不变
- 领域不变
- Tooltip 结构基本不变
- 详情和评分主流程不变

#### 复用

- 复用现有 `recommendationScore`
- 复用现有书籍主数据更新能力
- 复用现有评分结果数据
- 复用当前首页雷达组件结构
- 复用当前领域色与交互态体系

### 12.12 风险与最小验收标准

- 主要风险：
  - 视觉表达过强，破坏雷达整体秩序
  - 高分态、Hover、选中态、最近更新态之间发生冲突
  - 颜色强度过重，削弱领域色分类作用
  - 最近更新态停留时长不合适，影响理解

- 建议默认实现细节：
  - 最近更新态持续约 `1000ms`
  - 一次只标记最后一本刚更新的书

- 最小可用版本验收标准：
  - 评分成功后，首页雷达中的对应书籍视觉权重立即变化
  - 变化至少包括：
    - 圆点大小
    - 光晕 / 描边
    - 颜色强度
  - 变化不影响扇区归属
  - 变化不影响圈层归属
  - Hover 与选中态仍正常工作
  - 评分结果页能明确提示“雷达已同步更新”
